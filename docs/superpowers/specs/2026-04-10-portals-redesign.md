# Типизированные порталы с подписями — дизайн

**Дата:** 2026-04-10
**Статус:** Утверждён, готов к плану реализации
**Автор:** Руслан (+ brainstorming с Claude Code)

## Проблема

В «Хрониках Эльдории» все порталы визуально идентичны — фиолетовые пульсирующие кольца, отрисованные единой функцией `drawPortalTile`. В одной только деревне их **шесть** (в открытый мир, в лес, в королевство, в подземелье, в адскую яму, на арену), и Вова (целевая аудитория, 11-14 лет) путается: невозможно запомнить, какой из них куда ведёт. Та же проблема в других картах, где порталы ведут обратно в деревню и в соседние локации.

**Дополнительная находка при ревью:** порталы в static maps сейчас **вообще не анимируются**. Функция `drawPortalTile` объявлена с параметром `frame=0`, но вызывается внутри `cacheMap()` один раз при загрузке — и кадр навсегда остаётся нулевым. Пульсация есть в коде, но не на экране. Это пропавшая фича, которую наш рефакторинг заодно восстановит.

## Цели

1. Каждый портал должен быть **узнаваем с расстояния** по цвету и центральному символу — игрок видит цветное пятно и уже понимает, куда оно ведёт.
2. При подходе (≤ 2 тайла) над порталом появляется **текстовая подпись** с названием локации назначения.
3. Покрытие — **все** порталы игры: в static maps, возвратные порталы в деревню, порталы в open-world структурах (village_portal на чанке (0,0) + cave-entrances в biome-структурах, ведущие в процедурные данжны).
4. Попутно восстановить анимацию пульсации, которую мы сломали архитектурой кеширования рендера.

## Non-goals

- Не трогаем логику телепортации (`checkPortals` в `map-loading.js`) — только рендер.
- Не трогаем иконки порталов на миникарте (`minimap.js`) и в fast-travel (`fasttravel.js`) — там своя система, работает отдельно.
- Не добавляем описания локаций / лор-тексты внутри плашки подписи (пока только название). Если захочется — отдельная задача.

## Архитектура решения

### Ключевая идея — выделить порталы в отдельный runtime-уровень рендера

Сейчас `drawPortalTile` вызывается внутри `cacheMap()` (`tilemap.js`) и `buildCanvas()` (`chunks.js`) — то есть портал «запекается» в offscreen canvas один раз при загрузке карты/чанка, без анимации и без знания о типе. Мы меняем это на:

1. **Tile-рендер (в кеше)** рисует под порталом только нейтральный фон (grass или stone floor), без колец и символов.
2. **Runtime-рендер** (каждый кадр) рисует порталы **поверх** blit'нутого кеш-canvas'а — с анимацией (`animFrame`) и типизацией (цвет колец и символ по `target`).

Это решает три задачи одним шагом:
- Возвращает анимацию пульсации.
- Делает type-specific рендер возможным (в момент runtime мы знаем `map.portals[]` и `target` каждого).
- Не ломает кеширование остальных тайлов — выигрыш производительности сохраняется.

Единый source-of-truth для типов порталов. Содержит:

### Изменения по файлам

| Файл | Что меняется |
|---|---|
| **`js/portals.js`** (новый, ~180 строк) | `PORTAL_STYLES`, `getPortalStyle`, `getNearbyPortal`, `drawPortalSymbol` с 11 pixel-art символами 8x8 (10 целевых + 1 fallback `question`) |
| `js/sprites.js` | `drawPortalTile(ctx, x, y, frame, target)` — принимает `target`, первым делом рисует `drawGrassTile` как нейтральную базу, затем кольца цветом из `getPortalStyle` + центральный символ через `drawPortalSymbol`. `tileDrawers[TILE.PORTAL]` → **no-op** (не рисует ничего) — портал в кеше не присутствует вообще, runtime-рендер рисует его полностью |
| `js/tilemap.js` | `renderMap` после blit'а cached canvas'а итерирует `map.portals[]`, для каждого портала — `drawPortalTile(ctx, screenX, screenY, animFrame, portal.target)`. Screen-coord'ы = `portal.col * TS - camera.x`, clipped к экрану |
| `js/chunks.js` | `getChunkCanvas` / `buildCanvas` — tileDrawers[TILE.PORTAL] no-op тоже применяется (поскольку та же функция используется). `getLoadedChunks()` **расширить**: возвращать `{cx, cy, canvas, worldX, worldY, tiles}` — сейчас возвращает без `tiles`, а нам нужен доступ для сканирования портал-координат в runtime-рендере |
| `js/tilemap.js` → `renderOpenWorld` | После blit'а каждого чанка — сканирует `chunk.tiles` на TILE.PORTAL-координаты (чанк 30×20 = 600 тайлов, сканирование линейное, <1ms). Для каждой найденной координаты определяет target: если `chunk.cx === 0 && chunk.cy === 0` → `'village'`, иначе → `'dungeon_ow'`. Рисует поверх |
| `js/rendering.js` → `renderPlay` | После `renderCompanions` + перед overlay'ами — вызывает `getNearbyPortal(map.portals, p.x, p.y, 2)`; если найден → рисует подпись над порталом: полупрозрачная плашка `rgba(0,0,0,0.75)` с обводкой цветом `ringColor` и жёлтым текстом `8px "Press Start 2P"`. Для open-world аналогично — сканирует PORTAL-тайлы в радиусе 2 через `chunkManager.getTileAtWorld(col, row)`, target определяется по `pixelToChunk` (чанк (0,0) → 'village', иначе → 'dungeon_ow') |
| `js/map-loading.js` | **Не меняется.** Логика `checkPortals` и телепортация остаются как есть — они уже работают через `target`, и мы ничего не ломаем |

### Pixel-art символы (8×8 квадрат в центре портала)

Все символы рисуются в квадрате 8×8, расположенном в центре тайла 32×32 (то есть при `x=12, y=12`). Базовый подход — `fillRect` на отдельных «пикселях» 1×1 с цветом, совпадающим с `ringColor` (или более светлым оттенком для контраста).

| Символ | Дизайн |
|---|---|
| `house` | Треугольная крыша + квадратный корпус + дверь по центру |
| `compass` | Ромб 4 точки по сторонам света (N/S/E/W) |
| `tree` | Треугольная крона + ствол 2×2 |
| `cliffs` | Три неровных зубца разной высоты |
| `stalactite` | Три сталактита сверху, три сталагмита снизу |
| `tower` | Прямоугольник + зубчатая вершина (3 зубца) |
| `crown` | Три зубца с кружками на вершинах |
| `skull` | Овал черепа + два глаза-впадины |
| `skull_stars` | Skull + 2 мерцающие точки по бокам |
| `flame` | Волнистые языки пламени |
| `swords` | Два меча крест-накрест |
| `question` (fallback) | Знак `?` |

### Плашка подписи

- Позиция: над порталом (`portal.col * TS + 16 - camera.x, portal.row * TS - 12 - camera.y`)
- Размер: авто по ширине текста + 8 px padding, высота 14 px
- Фон: `rgba(0,0,0,0.75)`, 1px обводка цветом `ringColor` портала (дополнительная визуальная связка)
- Текст: `8px "Press Start 2P"`, цвет `#ffd54f` (золотой, согласно стилю других подсказок в игре — `[H] Справка`, `[E] Говорить`)
- Появление: мгновенное при входе в радиус 2 тайла, исчезновение — мгновенное при выходе. Без fade для простоты (fade можно добавить позже отдельной задачей).

## Риски и edge-cases

### 1. Конфликт плашки с другими overlay'ами
В `renderPlay` уже рисуются подсказки `[E] Говорить` (возле NPC) и `[H] Справка` (мигающая в деревне на уровне 1). Наша плашка рисуется над порталом, не над игроком — не должна пересекаться с ними. **Проверка при тесте:** стоит ли одновременно видеть `[E] Говорить` и подпись портала, если NPC возле портала? Должна. Если плашка закрывает NPC promt — сместить выше.

### 2. `dungeon_ow` не существует в исходных данных
У порталов в open-world структурах нет поля `target='dungeon_ow'` в каком-либо файле — это маркер, который мы присваиваем только на этапе рендера по координате чанка. Не записываем в save, не меняем данные структур. Чисто rendering-only.

### 3. Fallback на неизвестный target
Если кто-то добавит новый target, например `ice_palace`, и забудет обновить `PORTAL_STYLES` — портал отрисуется фиолетовыми кольцами с символом `?`. Это **явный визуальный баг-сигнал**, лучше чем тихий сбой.

### 4. Grass-фон под порталом в dungeon/cave
Поскольку `drawPortalTile` всегда рисует `drawGrassTile` как базу (упрощение: не знаем контекст из вызова), в картах dungeon/cave/hellpit, где пол — камень/лава, по краям портала (вне кольца) будет виден небольшой участок травы. **Приемлемо как trade-off** для простоты: магический портал и так аномалия в окружении. Если в playtesting'е будет диссонанс, добавляется параметр `baseTile` в `PORTAL_STYLES` и выбирается по target.

### 5. Производительность
- Static map: 6-10 порталов × 1 вызов `drawPortalTile` на кадр = ~60 fillRect-ов. Игнорируемо.
- Open world: в кадре ~9 чанков (3x3), каждый сканируется на PORTAL-тайлы (600 тайлов). 9 × 600 = 5400 проверок целых чисел на кадр = <0.1ms. Можно кешировать список PORTAL-координат при build'е чанка, но это преждевременная оптимизация.

### 6. Portal pooling
Если несколько порталов близки друг к другу (в деревне row 18 — три портала рядом), плашки могут перекрываться. `getNearbyPortal` возвращает **ближайший** — значит показывается только одна плашка. Это приемлемо: игрок физически стоит рядом с одним порталом, остальные различаются визуально (цветом колец).

### 7. `village_portal` структура в open-world
Структура `village_portal` на чанке (0,0) содержит тайл `TILE.PORTAL`. Наш сканер найдёт его и пометит как `'village'`. Лейбл «В деревню». ✓

## Стратегия тестирования

Регрессионный визуальный smoke-test — обязательно пройти по всем картам с порталами:

1. **Village** → 6 порталов (openworld/forest/kingdom/dungeon/hellpit/arena) должны иметь разные цвета/символы/подписи.
2. **Forest** → возврат в деревню (голубой, «В деревню») + портал в canyon (охра, «Ущелье»).
3. **Canyon** → forest + cave.
4. **Cave** → canyon + castle.
5. **Castle** → cave.
6. **Kingdom** → возврат в village.
7. **Hellpit** → возврат в village.
8. **Arena** → возврат в village.
9. **Dungeon** (статический через портал из village) → визуал «Подземелье» при входе.
10. **Open-world**: village_portal на (0,0) — голубой домик «В деревню». Пещерный портал в любой biome-структуре — тёмно-серый череп «Глубины».

**Негативный тест**: логика телепортации не должна сломаться — нажимаешь портал, переносишь, возвращаешься, всё работает.

**Regression для анимации**: должна визуально пульсировать в **static maps** (раньше не пульсировала!).

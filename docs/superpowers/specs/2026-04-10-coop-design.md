# Кооп-режим "Drop-in герой" — Спецификация

**Статус:** Дизайн. Реализация после выполнения `plans/2026-04-10-pre-coop-refactor-and-mobile.md`.

**Ревизия 2026-04-10 (update):** уточнены 5 ключевых решений по результатам повторного брейншторма:
1. Камера — midpoint между игроками (не за хостом)
2. Код комнаты — 6 символов (не 4)
3. Гость **приносит своего персонажа** (класс, уровень, оружие, броня, инвентарь) — не фиксированный knight
4. Гостевой сейв — отдельный слот `eldoria_save_guest_character`, обновляется после сессии (не эфемерен)
5. Сессия 4 добавляет ссылку-инвайт `?join=CODE`

## Цель

Добавить локальный/удалённый кооп для двух игроков: один игрок создаёт комнату (код из 4 символов), второй присоединяется и играет за второго героя в том же мире. Host-authoritative архитектура: вся логика крутится у хоста, гость только отправляет свой ввод и рендерит присланное состояние.

**Целевая аудитория:** Vova (11-14 лет) играет с другом через интернет или в той же сети. Публичный сервис не строим — никаких матчмейкингов, лидербордов, модерации.

**Объём:** ~1100 строк нового кода, 3-4 сессии разработки. Спека покрывает только первую рабочую версию (MVP), дальнейшие улучшения — в отдельных спеках.

## Роли

### Host (создатель комнаты)
- Запускает свой gameLoop как в одиночной игре
- Держит полный `game` state (players, enemies, projectiles, boss, chunks, worldSeed)
- Получает input от гостя через WebSocket, применяет к `players[1]`
- Рассылает snapshot всем подключённым клиентам (пока только один гость)
- Сохраняет прогресс в localStorage (как обычно)
- При дисконнекте гостя — продолжает играть один

### Guest (подключившийся)
- Подключается по коду комнаты
- **Приносит своего персонажа** (class/level/weapon/armor/ownedWeapons/ownedArmor/artifacts/potions/coins) через `{t:"hello"}` при handshake. См. раздел "Drop-in".
- **Не крутит** собственный game loop с физикой мира
- Его game loop содержит только: captureLocalInput → sendInput → receiveSnapshot → interpolate → render
- Пишет в `players[1].input`, отправляет на хоста 30 раз в секунду
- Получает snapshot'ы 20 раз в секунду, перезаписывает локальный `game` state
- **Персонаж** гостя сохраняется в отдельный слот `eldoria_save_guest_character` после завершения сессии. **Прогресс мира** (квесты, убитые боссы, открытый мир) гостем не сохраняется — это мир хоста. См. раздел "Save/load".

### Relay (WebSocket сервер)
- **Stateless** room manager: `Map<roomCode, Set<WebSocket>>`
- Никогда не парсит содержимое сообщений, не хранит game state
- Только forwardит бинарные/текстовые сообщения между клиентами комнаты
- Auto-expire комнаты через 30 минут неактивности (таймер сбрасывается при любом сообщении)
- Подъём: `node server/relay.js` в Docker-контейнере, listen на 8080

## Транспорт

- **WebSocket** через nginx `/ws` → `relay:8080` (docker internal network)
- URL клиента: `wss://eldo.evgosyan.ru/ws` в prod, `ws://localhost:3040/ws` в dev
- Cloudflare: WebSockets включён (проверить в панели перед деплоем relay)
- Формат сообщений: **JSON** (UTF-8 text frames). Бинарный протокол — оптимизация на будущее, не сейчас.
- Размер snapshot: цель ≤ 4 KB. Если больше — резать particles/далёких projectiles.

## Room Code

- **6 символов** из алфавита `ABCDEFGHJKLMNPQRSTUVWXYZ23456789` (32 символа — без похожих O/0, I/1)
- Пространство: 32^6 ≈ 1.07 млрд комбинаций — коллизии практически невозможны
- Генерируется на сервере при `createRoom`. При коллизии (комната с таким кодом существует) — регенерация
- Показывается хосту на экране, он диктует другу голосом или отправляет через мессенджер
- **Ссылка-инвайт (Сессия 4):** хост также видит кнопку "Скопировать ссылку" — URL вида `https://eldo.evgosyan.ru/?join=XKCD42`. Друг открывает → парсинг URL в `lobby.js` → автозаход в комнату
- Живёт пока хост подключён + 30 мин grace period после отключения хоста

## Протокол сообщений

### Клиент → Relay

```json
{ "type": "createRoom" }
```
Ответ: `{ "type": "roomCreated", "code": "XKCD42", "role": "host" }`
Или: `{ "type": "error", "reason": "serverFull" }` (при > 50 одновременных комнат)

```json
{ "type": "joinRoom", "code": "XKCD42" }
```
Ответ: `{ "type": "joined", "role": "guest" }` + запускается forward режим
Или: `{ "type": "error", "reason": "notFound" }` / `"roomFull"` (уже 2 игрока) / `"expired"`

```json
{ "type": "leave" }
```
Relay закрывает сокет клиента, отправляет остальным `{ "type": "peerLeft" }`

### Host ↔ Guest (через relay)

**Guest → Host (input, 30 Hz):**
```json
{
  "type": "input",
  "seq": 1234,
  "dx": 0.6,
  "dy": -0.8,
  "buttons": {
    "attack": true,
    "interact": false,
    "potion": false,
    "ability1": false,
    "ability2": false,
    "ability3": false
  },
  "edges": {
    "inventoryToggle": false,
    "menuToggle": false,
    "questsToggle": false
  }
}
```
- `seq` — монотонный счётчик, хост игнорирует устаревшие (если пришёл seq ниже последнего обработанного)
- `dx/dy` — нормализованные -1..1
- `buttons` — held-состояния (включены пока кнопка зажата)
- `edges` — одноразовые триггеры (истинны только в кадре нажатия)

### Служебные

```json
{ "type": "ping", "ts": 1728567890000 }
```
Клиент отправляет раз в 5 сек. Хост/гость отвечает `{ "type": "pong", "ts": ... }`. Нужно для latency overlay и detection "заснувшей" комнаты.

```json
{ "type": "chat", "text": "идём к пещере" }
```
Опционально на будущее. На первом этапе — не реализуем.

```json
{ "type": "kicked", "reason": "hostLeft" }
```
Отправляется хостом гостю при явном выходе хоста. Гость возвращается в главное меню.

## Snapshot Content — приоритет экономии байт

В порядке важности (что резать при превышении 4 KB):

1. **Обязательно:** players, boss, ближайшие ко 2 игрокам 20 врагов
2. **Важно:** projectiles (≤ 30 ближайших)
3. **Средне:** events (≤ 10, остальные терять)
4. **Можно резать:** далёкие враги (> 400px от обоих игроков), particles (вообще не передаются — гость генерирует свои локально)

**Не передаётся никогда:** tilemap (гость генерирует из `worldSeed`), chunks detailed state, NPC диалоги (гость не взаимодействует с NPC в MVP), inventory (только у хоста).

## Частоты и интерполяция

- **Input:** 30 Hz (33.3 мс) — достаточно для RPG-темпа, экономит batter
- **Snapshot:** 20 Hz (50 мс) — компромисс между трафиком и плавностью
- **Render:** 60 Hz как обычно
- **Interpolation:** гость рендерит с задержкой 100 мс (2 snapshot'а), линейно интерполируя между двумя последними позициями. Это маскирует jitter сети.

Пример буфера гостя:
```
T=0ms    snapshot #1 (pos=100)
T=50ms   snapshot #2 (pos=104)
T=100ms  snapshot #3 (pos=108)  ← рендерим позицию между #1 и #2 (pos=102)
T=150ms  snapshot #4 (pos=112)  ← рендерим между #2 и #3 (pos=106)
```

**Экстраполяция:** если snapshot потерялся и следующий не пришёл к T+100мс — продолжаем двигаться по последнему вектору скорости до 200мс задержки, затем фризим.

## Авторитет и конфликты

**Хост решает всё:**
- Урон (кто кого ударил, сколько отнял)
- Смерть и respawn
- Лут (что выпало, кто подобрал)
- Переход между чанками
- Спавн врагов
- Триггеры событий (сундуки, нашествия, боссы)

**Гость только:**
- Двигает своего игрока (хост валидирует коллизии)
- Решает атаковать / использовать способность (хост применяет урон)
- Рендерит UI локально (HP bars, minimap)
- Локальные эффекты: particles, haptics, sfx

**Конфликты (например, оба игрока подобрали один сундук одновременно):** хост обрабатывает в порядке прихода input'ов, второй получает snapshot где сундука уже нет.

## Камера при двух игроках

**MVP решение:** камера следует за **midpoint** между игроками. `camera.followMidpoint(p1, p2)`:
- Если `players.length === 1` (SP) — fallback: центр единственного игрока (существующее поведение)
- Если `players.length === 2` (коопа) — центр = `((p1.x + p2.x) / 2, (p1.y + p2.y) / 2)`
- Границы камеры фиксированы существующей логикой клампа к карте

**Принудительная телепортация:** если `distance(p0, p1) > TELEPORT_LIMIT` (≈ 600px, настраивается), немедленно телепортируется **гость** к хосту (`p2.x = p1.x + 40, p2.y = p1.y`). Это гарантирует что гость всегда в пределах одной камеры и в подгруженных чанках хоста. Никакого grace period — телепорт моментальный, чтобы избежать состояний "гость в неподгруженном чанке".

**Не делаем в MVP:** split-screen, dynamic zoom, отдельная камера гостя. Можно рассмотреть в будущей итерации.

## Drop-in и reconnect

**Drop-in:** гость может подключиться к комнате в любой момент (даже во время боя). Последовательность:
1. Гость открывает lobby → вводит код → WS connect → `{t:"joinRoom", code}`
2. Relay форвардит хосту `{t:"peerJoined"}`
3. Гость шлёт `{t:"hello", character:{...}}` — см. ниже
4. Хост создаёт `players[1]` с переданными статами, помещает рядом с `players[0]` (+40px справа, или в безопасной точке если хост в бою)
5. Хост отправляет `snapshot` с `full: true` (полное состояние мира)
6. Гость начинает рендерить интерполяцию snapshot'ов

**Гость приносит своего персонажа.** В `{t:"hello"}` передаются поля из гостевого сейва:
```json
{
  "t": "hello",
  "character": {
    "class": "archer",
    "level": 7,
    "hp": 130, "maxHp": 130,
    "atk": 18,
    "weapon": "longbow",
    "ownedWeapons": ["iron_sword","longbow","crossbow"],
    "equippedArmor": {"helmet":"leather_cap","chest":"leather_chest","legs":"leather_legs","shield":null},
    "ownedArmor": ["leather_cap","leather_chest","leather_legs"],
    "artifacts": {"earth":false,"fire":true,"water":false},
    "potions": 5,
    "coins": 340
  }
}
```

- Хост применяет эти значения к `players[1]` — это его "рабочая копия" персонажа гостя
- XP/монеты/лут, заработанные в сессии, хост периодически шлёт обратно гостю через `{t:"character_sync", xp, coins, level, hp, newWeapons:[...]}`
- **Убитые боссы засчитываются только хосту** (это его мир и его прогресс). Гость НЕ помечает их как убитых в своём сейве. Квесты тоже только у хоста.
- При явном выходе или дисконнекте гость сохраняет обновлённый character в `eldoria_save_guest_character` (см. раздел Save/load)

**Если гость не имеет сохранения** (первая игра, сейва нет): lobby предлагает "Быстрый старт — Knight, уровень 1". Это fallback-поведение из старой спеки.

**Reconnect после дисконнекта:** НЕ реализуется в MVP. Порвалось — комната умерла, делаем новую.

## Смерть игроков

**Смерть гостя:** respawn через 3 сек рядом с хостом, HP = 50% max. Хост не должен ждать гостя — игра продолжается.

**Смерть хоста:** game over для обоих, возврат в главное меню. Гостю приходит `{ type: "kicked", reason: "hostDied" }`.

**Смерть обоих одновременно:** хост обрабатывает в своём gameLoop, оба видят экран game over.

## Save/load

Два независимых сейва в localStorage:

- **`eldoria_save`** (существующий) — полный сейв мира + персонаж **хоста**. Квесты, убитые боссы, открытый мир (seed, changes, kills), инвентарь, локация. Работает как в SP.
- **`eldoria_save_guest_character`** (новый) — **только персонаж** гостя (class, level, hp/maxHp, atk, weapon, ownedWeapons, equippedArmor, ownedArmor, artifacts, potions, coins). Ни квестов, ни мира, ни убитых боссов, ни локации. Обновляется в двух случаях:
  1. Когда игрок запускает SP из главного меню и у него есть персонаж в этом слоте (этот слот можно использовать как "второй SP-персонаж" опционально)
  2. В конце кооп-сессии (при дисконнекте или явном выходе) — актуальные xp/coins/level/hp/newWeapons применяются к слоту и сохраняются

**Главное меню хоста:** кнопка "Продолжить" грузит `eldoria_save` (как сейчас).

**Главное меню гостя:** при нажатии "Кооп → Присоединиться" lobby подтягивает `eldoria_save_guest_character` если он есть и шлёт в `{t:"hello"}`. Если слота нет — fallback на дефолтного knight lvl 1.

**Загрузка хостом сохранения во время кооп:** не поддерживается. Загрузить сейв можно только из главного меню (до старта комнаты).

**Приоритет:** гость **не может** принести своего "хостового" персонажа (с убитыми боссами, квестами, открытым миром). Приносится только базовый character без world-state. Это by design — миры не сливаются.

## Безопасность и privacy

- **Целевая аудитория:** дети играют с друзьями. Публично ничего не светим.
- **Никакой валидации input'а:** гость может прислать `dx=100` — хост клэмпит на [-1, 1]
- **Cheat prevention:** не нужен. Играют с друзьями. Если гость "читит" — хост видит и выкидывает через `kick`.
- **Нет чата:** нет модерации, нет токсичности
- **Никакого логирования:** relay не пишет в файлы, game state нигде не персистится кроме localStorage хоста
- **Auto-expire:** комнаты умирают через 30 мин неактивности, чтобы relay не копил зомби
- **Rate limit:** relay ограничивает входящие сообщения 100 req/sec на сокет — защита от runaway-клиента

## Cloudflare и nginx

**Cloudflare:**
- WebSockets должны быть включены в панели для `eldo.evgosyan.ru` (по умолчанию ON в Free plan, проверить)
- Timeout соединений Cloudflare Free: 100 сек idle — наш `ping` раз в 5 сек удерживает соединение живым
- Если CF начнёт обрубать — временно отключить proxy для поддомена (grey cloud)

**nginx (будет раскомментирован из `nginx.conf`):**
```nginx
location /ws {
  proxy_pass http://relay:8080;
  proxy_http_version 1.1;
  proxy_set_header Upgrade $http_upgrade;
  proxy_set_header Connection "upgrade";
  proxy_set_header Host $host;
  proxy_set_header X-Real-IP $remote_addr;
  proxy_read_timeout 3600s;
  proxy_send_timeout 3600s;
}
```

## Граничные случаи

| Случай | Обработка |
|--------|-----------|
| Потеря snapshot | Гость экстраполирует до 200 мс, затем фриз |
| Потеря input | Хост использует последний известный input пока не придёт новый |
| Одновременный урон по врагу | Хост применяет в порядке прихода, итоговое HP в следующем snapshot |
| Гость подключается во время cutscene/диалога | Хост отправляет snapshot, но гость замораживается пока хост в диалоге |
| Оба в разных чанках | Немедленный телепорт гостя к хосту при `distance > TELEPORT_LIMIT` (≈ 600px) |
| Гость спамит input | Relay rate-limit 100/sec, хост игнорит > 30/sec |
| Relay упал | Клиенты получают WebSocket close, показывают "соединение потеряно", возврат в меню |
| Cloudflare обрубил по idle | Ping каждые 5 сек предотвращает, но если упало — то же что relay упал |
| Гость в iPhone Safari, а WS не работает | Fallback: сообщение "браузер не поддерживается" в меню create/join. Safari 16+ должен работать |

## Решения (ревизия 2026-04-10)

1. ~~**Выбор класса для гостя:**~~ **РЕШЕНО:** гость приносит своего персонажа (класс, уровень, оружие, броня) через `{t:"hello", character:{...}}`. Fallback на knight lvl 1 если сейва нет. См. раздел "Drop-in".
2. **Голосовая связь:** не делаем. Используют Discord/телеграм.
3. **Общий инвентарь или раздельный:** раздельный. Гость не видит инвентарь хоста.
4. **XP sharing:** оба получают одинаковое количество XP от каждого убийства (хост рассчитывает). Гость получает через `{t:"character_sync"}`.
5. **Лут:** первый подобравший забирает. Никакой round-robin. Новое оружие для гостя — через `character_sync.newWeapons`.
6. **Квесты:** в MVP квесты работают только у хоста. Гость помогает убивать, но квесты не получает.
7. **Убитые боссы:** засчитываются только хосту в его `defeatedBosses`. Гость в своём SP по-прежнему должен их убить сам.
8. **Камера:** midpoint между игроками, телепорт гостя при `distance > TELEPORT_LIMIT`.
9. **Код комнаты:** 6 символов (не 4) — из соображений минимального риска коллизий при 50 одновременных комнатах.
10. **Ссылка-инвайт:** добавляется в Сессии 4 (полировка) как `?join=CODE`.

## Масштаб и Out of Scope

**В MVP (coop-session1..3, ~1100 строк):**

- Интерполяция гостя
- Телепорт гостя к хосту
- Respawn гостя

- Voice chat

- Комната создаётся за < 2 секунды после клика "Создать"
